# triary 要件整理

## 背景・目的

筋トレの記録と漸進性過負荷の振り返りを目的とした個人向け PWA。
プログラミング学習も兼ねた個人プロジェクトだが、成果物は public に公開する
（広報はしない／知人が偶然たどり着いて使う程度を許容する）。

- **誰が**: 作者本人がメインユーザー。public 公開のため不特定の他者も
  サインアップ可能
- **なぜ**: トレーニング内容を継続的に記録し、後から重量・回数の推移や
  カリステニクスのスキル習得段階を振り返って、漸進性過負荷の原則を
  実践したい。ウエイトの「重量を伸ばす」だけでなく、自重種目の
  「より難度の高いバリエーションへ進む」進捗も同じ仕組みで扱う
- **個人情報ポリシー**: 個人情報は一切収集しない（メールアドレスも不要）
- **配信形態**: PWA。ストア審査を回避しつつスマホ UX を担保する

## スコープ

### Must Have（MVP / In Scope）

トレーニングを記録し、後から見返せる最低限のループを成立させる。

1. ユーザー登録・ログイン（任意のユーザー ID + パスワード）
2. 種目マスタの参照（プリセット）と種目の追加・編集・削除
3. トレーニングセッションの明示的な開始・終了とその下での記録入力
4. セッション履歴の閲覧（日付別ビュー）
5. 種目別の履歴閲覧（前回の重量・回数を確認できる粒度）
6. 記録の編集・削除

### Should Have（次フェーズ）

- スコアリング: 重量・回数・セット数などセッションの構成要素を集計し、
  推移を可視化する。各要素には **ユーザーが調整できる係数** を掛けて
  スコアを算出し、「回数より重量の伸びを重視したい」といった比重の
  コントロールができるようにする（係数のプリセットも用意する想定）
- 種目別／期間別の集計とグラフ
- セット間インターバルの記録支援（タイマー）

### Could Have / Future Consideration

- ワークロードの事前設定と「まとめて種目」（複数種目を 1 ワークロードとして
  テンプレ化する機能）
- RPE などの主観強度記録
- 推定 1RM・別スコアモデル

### Out of Scope

- ソーシャル機能（フォロー、いいね、共有）
- 個人情報・連絡先の収集
- ネイティブアプリ（iOS / Android ストア配信）
- オフライン書き込み・バックグラウンド同期
- 食事・体重・睡眠などのライフログ
- ソフトデリート／ゴミ箱（誤削除はユーザーの自己責任）

## 機能要件

### F1. アカウント

- F1.1 ユーザーは任意の文字列の **ユーザー ID** とパスワードでサインアップできる
- F1.2 ユーザー ID は全ユーザー間で一意
- F1.3 ログイン後はセッションが維持され、ログアウトできる
- F1.4 パスワード変更ができる
- F1.5 メールアドレス等の個人情報は一切要求しない
- F1.6 パスワードリセット機構は MVP では提供しない（紛失 = アカウント喪失）

### F2. 種目マスタ

- F2.1 システム提供のプリセット種目が初期状態で参照可能
- F2.2 ユーザーは自分専用の種目を追加・編集・削除できる
- F2.3 種目はプリセットでもユーザー作成でも、**1 つ以上の計測項目
  (measurement kind)** を持つ。MVP で扱う計測項目は以下：
  - `reps`: 回数（整数）
  - `weight`: 重量（kg、符号付き。正 = 加重、負 = 補助／アシスト）
  - `time`: 秒数（整数）
- F2.4 計測項目はそれぞれ「必須 / 任意」を種目側で指定する。例：
  - ベンチプレス: `reps` 必須、`weight` 必須
  - 懸垂: `reps` 必須、`weight` 任意（+ で加重ベルト、− でアシスト）
  - プランク: `time` 必須
  - 加重プランク: `time` 必須、`weight` 任意
- F2.5 計測項目の組み合わせによって、後段のセット記録で入力できる
  フィールドが変わる
- F2.6 プリセット種目は編集・削除不可（自分用に複製は可）
- F2.7 種目は **進捗ツリー（プログレッション）** を構成できる。ある種目を
  「目標種目（ゴール）」として、それに到達するための易しいバリエーションを
  子種目としてぶら下げられる
  - 例: ゴール = ハンドスタンドプッシュアップ。子 = パイクプッシュアップ、
    デクラインパイクプッシュアップ、ウォールアシステッド HSPU など
  - 「ネガティブ懸垂」のように動作そのものを変える派生も、別の種目として
    親 (= 通常の懸垂) の子に位置づける
  - 子種目はさらに自分の子を持てる（n 段ネスト可）。1 つの子は 1 つの親に
    だけ属する（ツリー構造、DAG ではない）
  - 親と子は同じ計測項目セットを持つ必要はない（例: ゴールが reps + weight
    でも、補強段階で time 系の保持系が混じってよい）
- F2.8 プログレッションの並びには順序（易 → 難）を持たせ、ユーザーが編集できる
- F2.9 ツリーの単独ノード（子を持たない種目）は、F2.7 の意味では「ゴール
  でもありプログレッション段階でもない、ただの種目」として扱う

### F3. セッションと記録

- F3.1 セッションは **ユーザーが明示的に開始する**。「トレーニング開始」
  操作で新しいセッションが作られる。日単位での自動グルーピングはしない
- F3.2 セッションは以下の属性を持つ：
  - `workout_date` (DATE): そのトレーニングが「いつのものか」を表す論理日付。
    DB には DATE 型として永続化する
  - `started_at` (TIMESTAMP): セッション開始の物理時刻
  - `ended_at` (TIMESTAMP, 任意): セッション終了時刻。終了操作をしないと NULL
  - `note` (任意): 自由記述メモ
- F3.3 セッション作成時、`workout_date` のデフォルトは「クライアントの
  ローカルタイムゾーンにおける `started_at` の日付」とする。ユーザーは
  これを上書き編集できる（バックデート登録対応）
- F3.4 同じ `workout_date` に複数のセッションを作ってよい
  （二部練・朝練 / 夜練の区別が自然にできる）
- F3.5 ユーザーは「このセッションに種目を追加」する形で記録を入力する。
  1 つの記録は「種目 × 順序付きセット列」で表現される（= 種目ブロック）
- F3.6 セットごとに記録できるフィールドは、種目に設定された **計測項目セット**
  に従う：
  - `reps` を持つ種目: 回数を記録
  - `weight` を持つ種目: 重量 (kg) を記録（符号付き。+ で加重、− で補助）
  - `time` を持つ種目: 秒数を記録
  - 上記いずれにも、（任意）セット間インターバル秒数を付けられる
  - 種目の必須計測項目はセット保存時に必須、任意計測項目は省略可
- F3.7 セットは種目ブロック内で順序を持つ（ユーザーが入れ替え可能）
- F3.8 同一セッション内で同じ種目を複数回登録することも許可する
  （ピラミッドセット等、フォーマットを変えて再度行うケース）
- F3.9 記録は **プログレッションツリーの任意のノード** に対して付けられる。
  同じゴール（例: ハンドスタンドプッシュアップ）に向かう練習でも、その日に
  実際に行ったのが「パイクプッシュアップ」ならパイクプッシュアップに対して
  記録する。ゴール種目に直接記録することと、その下のバリエーションに
  記録することはどちらも等しく有効
- F3.10 セッション・種目ブロック・セットはそれぞれ編集・削除できる
- F3.11 削除は即時反映で復旧不可（確認ダイアログで警告する）

### F4. 履歴閲覧

- F4.1 **日付別ビュー**: カレンダー or 日付降順リストでセッションを一覧でき、
  選択するとそのセッションの全種目・全セットが見える
- F4.2 **種目別ビュー**: 種目を選ぶとその種目の過去セッションが新しい順に
  並び、各セッションでのセット内容が確認できる
- F4.3 種目別ビューでは「前回の数値」を一目で把握できる UI を優先する
  （次回トレーニング時に重量を決める動線になるため）
- F4.4 種目別ビューは **プログレッションツリー単位での集約閲覧** にも
  対応する。ゴール種目を選ぶと、そのゴール自身およびすべての子孫種目に
  対する記録が時系列で混ざった状態で見え、「どの段階でどれだけ積んできて、
  いつ次の段階に上がったか」が辿れること

## 非機能要件

### NF1. 利用環境・互換性

- スマートフォンのモダンブラウザ（iOS Safari、Android Chrome）を主軸に最適化
- デスクトップブラウザでも崩れず使えること
- PWA としてホーム画面追加・スタンドアロン表示に対応
- **オンライン前提**。オフライン中の書き込みはサポートしない

### NF2. パフォーマンス

- 同時利用者は実質的に数名想定。スケールアウト要件はない
- 1 セッションの記録入力〜保存は体感即時（< 500ms 程度）であること
- 履歴 API は直近のデータを優先返却し、ページネーションで古いものに辿り着ける

### NF3. セキュリティ

- パスワードは適切な KDF（argon2 / bcrypt 等）でハッシュ化して保存
- 自分のデータ以外には絶対にアクセスできない（行レベルで user_id 分離）
- HTTPS 前提
- public 公開のためサインアップフォームに何らかの濫用対策（レート制限程度）を入れる

### NF4. データ保全

- ソフトデリートなし。ユーザー削除操作は即時かつ復旧不可
- バックアップは運用側で取得するが、ユーザー向けの復元機能は提供しない
- 誤操作対策は「削除前の確認 UI」のみで担保する

### NF5. 開発・運用

- バックエンド: Rust + Axum + MySQL
- フロントエンド: TypeScript + SolidJS + Tailwind + CSS Modules
- マイグレーションは生 SQL（`sqlx-cli` 管理）
- API は OpenAPI スキーマファースト
- 統合テストは Postman / Newman で言語非依存に保つ
- フロントの lint/format は Biome のみ

## 前提条件

- 作者単独で開発・運用する。複数人開発・複数テナントの管理機能は不要
- ホスティング先・ドメイン・運用基盤は本要件の範囲外（別途検討）
- ユーザー数は一桁〜二桁前半を想定し、それを越える場合は別途見直す

## リスク・未決事項

| 項目 | 内容 | 対応方針 |
|---|---|---|
| 認証実装の詳細 | パスワード方式は決まったが、セッション管理（Cookie / JWT）、CSRF 対策、レート制限の具体は未決 | 設計フェーズで決定。ADR 候補 |
| スコアリング設計 | MVP では実装しないが、将来「重量・回数・セット数等に係数を掛けた重み付きスコア」を計算する。データモデルがこの集計を阻害しないようにしたい。係数の保持単位（ユーザー単位 / 種目単位 / 種目タイプ単位）も未決 | 設計フェーズで「Future の重み付き集計が破綻しないスキーマ」を意識する。係数スコープは Should Have 着手時に決定 |
| 計測項目の拡張 | 将来的に距離・心拍・RPE などを追加する可能性 | 計測項目を「種目に紐づくセット」として持ち、種目側で必須/任意を選べる構造にする。新項目追加が既存記録を壊さないこと |
| 加重と補助の表現 | 符号付き `weight` 1 フィールドで「+ 加重 / − 補助」を表現する暫定方針。実際のトレーニングで「補助 0 と加重 0」「補助 + 加重」のような複合パターンが出ないか | MVP は符号付き 1 フィールドで開始。複合パターンが頻出するなら設計時に分離フィールド化を検討 |
| プログレッションツリーの設計 | 親子で計測項目セットが異なってよいので、ゴール単位の集計は単純な数値合算ができない（reps と秒数を足せない）。スコアリングや「前回値」表示と整合させる方法が未決 | 設計フェーズで、ツリー集約時の表示・集計ルールを決める。Should Have のスコアリング着手時には正式に詰める |
| タイムゾーン | `workout_date` はクライアントのローカル日付（DATE）として永続化、`started_at` / `ended_at` は UTC TIMESTAMP として永続化 | サーバーは TZ を一切持たない。フロントが `Intl` / `Date` を用いてブラウザの TZ で表示・入力デフォルトを吸収する。ユーザーがデバイスの TZ を変えればそれに追従する |
| ワークロード機能 | Future 行きだが、種目グルーピングを後付けする際にセッション設計と衝突しないか | データモデル設計時に Future を一度は試算しておく |
| public 公開時の濫用 | サインアップ無制限だとスパム登録の的になりうる | レート制限・IP ベース制限等を運用フェーズで検討 |
| パスワード紛失 | リセット機構なしのため、紛失 = アカウント喪失 | 利用規約相当の注記で明示。MVP の割り切りとして許容 |
