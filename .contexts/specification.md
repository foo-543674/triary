# triary 仕様書 (MVP)

`requirements.md` で整理した要件を、受け入れ基準を書ける粒度まで具体化したもの。
本仕様書は **MVP (Must Have)** のスコープに対応する。Should Have / Future
は対象外。

## 用語

| 用語 | 定義 |
|---|---|
| ユーザー | アカウントを持つ triary 利用者 |
| 種目 (Exercise) | トレーニングの 1 単位（例: ベンチプレス、パイクプッシュアップ） |
| 計測項目 (measurement kind) | 1 セットで記録する数値の種類。MVP では `reps` / `weight` / `time` の 3 種類 |
| 計測項目セット | 種目ごとに「どの計測項目を必須/任意で記録するか」を指定したもの |
| プログレッションツリー | 種目の親子関係。親は「ゴール」、子は「より易しいバリエーション」 |
| セッション | ユーザーが明示的に開始 / 終了するトレーニングの単位。論理日付 `workout_date` を持つ |
| 種目ブロック | あるセッションにおける、ある種目に対する 1 回分の取り組み（複数セットを含む） |
| セット | 種目ブロック内の 1 セット分の数値（種目の計測項目セットに従う） |

## 主要な前提（仕様レベルの決め）

これらは要件側で決まっていなかったが、MVP を仕様化するにあたり仮置きする。
設計フェーズで再検討する余地を残す。

| 項目 | 仮置き値 | 根拠 |
|---|---|---|
| セッションの境界 | ユーザーが「トレーニング開始」操作で明示的に作る。終了操作は任意（つけ忘れても OK） | 自動グルーピングだと二部練が表現できず、深夜跨ぎも壊れる |
| 論理日付 (`workout_date`) | DB には `DATE` 型で永続化。新規セッションのデフォルト値はフロントが `Intl` / `Date` でブラウザのローカル日付を計算してリクエストする。ユーザーが上書き編集可能 | 文字列ではなく型として扱い、範囲検索やソートを安全にする |
| 物理時刻 | `started_at` / `ended_at` は `TIMESTAMP` (UTC 保存)。表示時はフロントがブラウザ TZ で変換する | サーバーは TZ を一切持たない |
| タイムゾーンの責務 | 完全にフロントで吸収。サーバーはユーザーの TZ をプロファイル等にも保持しない | デバイス側の TZ 変更（旅行・引越）に自動追従できる |
| ユーザー ID 文字種 | 半角英数・ハイフン・アンダースコア。3〜32 文字 | URL に乗せられる安全な範囲 |
| ユーザー ID の予約語 | `admin`, `api`, `system` 等の最低限を予約（具体リストは設計時） | 衝突回避 |
| パスワード長 | 12 文字以上、128 文字以下 | 現代的最低水準 |
| 種目名 | 1〜64 文字、ユーザー内（プリセット含む）でユニーク | 表示と取り違え防止 |
| プリセット種目数 | MVP では数十件程度（自重・ウエイトの定番）を初期 SQL シードで投入 | スコープ制御 |
| プログレッションツリー深さ | 最大 8 段 | 実用上十分かつ UI で破綻しない |
| プログレッションツリー子数 | 1 親あたり最大 16 子 | 実用上十分 |
| 重量 | −500.0〜999.9 kg、0.1 刻み（負値 = 補助／アシスト、正値 = 加重） | 加重・補助を符号付き 1 フィールドで表現 |
| 回数 | 0〜9,999 回 | 上限想定 |
| 時間 | 0〜86,400 秒（24 時間） | 上限想定 |
| インターバル | 0〜3,600 秒 | 上限想定 |
| 計測項目セット | 種目ごとに `reps` / `weight` / `time` から 1 つ以上を選択。それぞれ「必須 / 任意」を指定 | 種目によって何を記録するかが大きく異なるため |
| 1 セッションあたりの種目ブロック数 | 上限なし（実用上 50 程度を想定） | 過度な制約は避ける |
| 1 種目ブロックあたりのセット数 | 最大 50 セット | 上限想定 |

## ユーザーストーリー

### US-1. アカウントを作成する

> As a 新規ユーザー,
> I want to ユーザー ID とパスワードを決めてサインアップする,
> so that triary に自分のトレーニングデータを保存できるようになる.

**受け入れ基準**:

- [ ] Given 未ログイン状態でサインアップ画面にいる, When 有効なユーザー ID
  と有効なパスワードを送信する, Then アカウントが作成され、自動的にログイン
  状態となり、メインのトレーニング画面に遷移する
- [ ] Given 既存ユーザーと同じユーザー ID で送信した, When サブミットする,
  Then エラー「このユーザー ID は使えません」が表示され、アカウントは作成
  されない（既存ユーザーの存在を露呈しないため、メッセージは「使えません」
  で統一する）
- [ ] Given ユーザー ID が 3 文字未満 / 33 文字以上 / 許可外文字を含む,
  When サブミットする, Then クライアントで弾かれサーバーには到達しない。
  クライアントを迂回した場合もサーバーが 400 で拒否する
- [ ] Given パスワードが 12 文字未満 / 129 文字以上, When サブミットする,
  Then 同様に拒否される
- [ ] Given 予約語 (`admin` 等) をユーザー ID として送信した, When サブミット
  する, Then 「このユーザー ID は使えません」エラー
- [ ] Given パスワードはハッシュ化されて保存される, When DB を直接覗いても
  平文のパスワードは存在しない

**境界条件**:

- ユーザー ID: 大文字小文字を区別する (`Foo` と `foo` は別) / 区別しない
  → **区別しない** に倒す（誤認防止）。保存時は小文字正規化。
- ユーザー ID 末尾・先頭の `_` `-`: 許可
- パスワードの文字種制約: 課さない（長さのみで担保）

### US-2. ログイン・ログアウトする

> As a 既存ユーザー,
> I want to ID とパスワードでログインし、終わったらログアウトする,
> so that 自分の端末以外でもデータを扱える / 共有端末で他人に見られない.

**受け入れ基準**:

- [ ] Given アカウントがある, When 正しい ID とパスワードを送信する,
  Then ログイン状態となり、メイン画面に遷移する
- [ ] Given アカウントがある, When 誤ったパスワードを送信する,
  Then 「ID またはパスワードが違います」というメッセージが表示される
  （ID 存在の有無は露呈しない）
- [ ] Given 存在しない ID を送信する, When サブミットする,
  Then 上と同じメッセージが返る
- [ ] Given ログイン状態, When ログアウトを実行する, Then セッションが破棄
  され、未ログイン画面に遷移する
- [ ] Given 同じアカウントで複数端末からログインする, When それぞれの端末
  で操作する, Then どちらの端末でも独立してセッションが維持される
  （MVP ではセッションの強制ログアウト機能は持たない）

**境界条件**:

- ログイン試行のレート制限: 同一 IP から 1 分間に 10 回まで（仮置き）。
  超えたら 429 を返す
- セッション有効期限: 30 日（仮置き）。設計フェーズで Cookie / JWT どちらか
  と合わせて確定する

### US-3. パスワードを変更する

> As a ログイン中のユーザー,
> I want to 現在のパスワードを確認した上で新しいパスワードに変更する,
> so that 漏洩懸念があるときに自衛できる.

**受け入れ基準**:

- [ ] Given ログイン中, When 現在のパスワードと新しいパスワードを送信する,
  Then パスワードが更新され、成功メッセージが表示される
- [ ] Given 現在のパスワードが間違っている, When サブミットする,
  Then エラーメッセージが表示され、パスワードは変更されない
- [ ] Given 新しいパスワードが長さ要件を満たさない, When サブミットする,
  Then エラーが返り変更されない
- [ ] Given パスワード変更が成功する, When 同じアカウントの他端末を見る,
  Then **MVP では既存セッションは無効化されない**（既知の割り切り。
  将来的に強制ログアウトを追加する余地は残す）

**境界条件**:

- 同じパスワードへの変更: 許可（サーバー側で旧と一致しても弾かない）

### US-4. プリセット種目を閲覧する

> As a 新規ユーザー,
> I want to サインアップ直後から定番種目のリストを参照できる,
> so that 何も登録しなくてもすぐにトレーニング記録を始められる.

**受け入れ基準**:

- [ ] Given 新規ユーザーが初回ログインする, When 種目一覧を開く,
  Then システム提供のプリセット種目（数十件）が表示される
- [ ] Given プリセット種目を選ぶ, When 詳細を見る, Then 種目名、種目タイプ、
  プログレッションツリー上の親子関係（あれば）が確認できる
- [ ] Given プリセット種目を編集または削除しようとする, When 操作する,
  Then 不可（ボタンが出ない／ API が 403 を返す）
- [ ] Given プリセット種目を「自分用に複製」する, When 操作する,
  Then 同じ内容のユーザー種目がコピーされ、編集可能になる

**境界条件**:

- プリセット種目はサーバー側 SQL シードで管理する。ユーザー側 DB 行は
  持たない（設計フェーズで詳細決定）

### US-5. 自分の種目を追加・編集・削除する

> As a ユーザー,
> I want to 自分のトレーニングに合った種目を自由に登録・調整・削除する,
> so that プリセットでは足りない種目も記録できる.

**受け入れ基準**:

- [ ] Given ログイン中, When 種目名・計測項目セット・（任意で）親種目を
  指定して種目を新規作成する, Then 自分専用の種目として保存され、種目一覧
  に出現する
- [ ] Given 計測項目を 1 つも選ばずに保存しようとする, When サブミットする,
  Then 「最低 1 つの計測項目が必要です」エラーで拒否される
- [ ] Given 種目名が空 / 64 文字超 / 同名の種目（プリセット含む）が既に
  自分のスコープに存在する, When サブミットする, Then 検証エラーで拒否される
- [ ] Given 自分の種目を選択した, When 種目名・計測項目セット・親種目を
  編集する, Then 変更が反映される
- [ ] Given 計測項目セットを変更する（項目の追加・削除・必須/任意の変更）,
  When 既にその種目に紐づく記録が存在する, Then **MVP では計測項目セットの
  変更を拒否する**（既存セットの解釈が食い違うため。割り切り）
- [ ] Given 計測項目セットを変更する, When まだ記録が存在しない,
  Then 自由に変更できる
- [ ] Given 自分の種目を削除しようとする, When その種目に紐づく記録が
  存在する, Then 警告ダイアログを表示し、確認後削除すると関連する種目
  ブロック・セットも **カスケード削除** される。復元不可
- [ ] Given 削除しようとする種目が他種目の親（プログレッション親）になって
  いる, When 削除する, Then 子種目は親が外れて孤立する（孤立は許可。
  カスケード削除はしない）

**境界条件**:

- 自分の種目とプリセットの名前衝突: 不可（同名禁止）
- 削除確認: 必ずモーダルで「この操作は取り消せません」と表示する

### US-6. プログレッションツリーを構築する

> As a カリステニクスを練習中のユーザー,
> I want to ゴール種目（例: ハンドスタンドプッシュアップ）の下に練習段階
> （パイク、デクライン、ウォールアシステッド…）をぶら下げて管理する,
> so that スキル進捗を 1 つのまとまりとして俯瞰・記録できる.

**受け入れ基準**:

- [ ] Given 種目 A が存在する, When 種目 A の親として種目 B を指定する,
  Then 種目 A は種目 B の子として登録され、ツリー上に表示される
- [ ] Given ツリーが既に存在する, When ある種目の親種目を別の種目に
  付け替える, Then ツリー構造が更新される（その種目とその子孫はまとめて
  移動する）
- [ ] Given 親種目を `null` に設定する, When 保存する, Then その種目は
  ルートになる
- [ ] Given 自分自身を親に指定する / 自分の子孫を親に指定する, When 保存
  する, Then 循環検出により拒否される
- [ ] Given ツリーの深さが 8 を超える設定をしようとする, When 保存する,
  Then 上限超過で拒否される
- [ ] Given 1 親あたりの子数が 16 を超える設定をしようとする, When 保存
  する, Then 上限超過で拒否される
- [ ] Given 親と子の計測項目セットが異なる, When 設定する, Then 許可される
  （例: ゴール `reps + weight` に対し補強段階 `time` の保持系を子にできる）
- [ ] Given 「ネガティブ懸垂」のように動作そのものを変えた派生を扱いたい,
  When 通常の懸垂の子として「ネガティブ懸垂」種目を作る, Then プログレッション
  ツリー上で正しく親子関係になり、両方に独立して記録できる
- [ ] Given プリセット種目を親に指定する, When 自分の種目を子として
  ぶら下げる, Then 許可される（プリセットゴールの自分用練習段階を作れる）

**境界条件**:

- 並び順: 兄弟間の順序はユーザーが手動で並び替えできる（ドラッグ or 上下
  ボタン）。デフォルトは作成順
- ツリーの可視化方法は仕様書では決めない（設計フェーズで選定）

### US-7. トレーニングセッションを開始して記録する

> As a ユーザー,
> I want to 「トレーニング開始」を押して新しいセッションを作り、種目を
> 選びながらセットを並べていく,
> so that 朝練 / 夜練・二部練を区別して記録でき、必要なら後でメモも残せる.

**受け入れ基準**:

- [ ] Given ログイン中, When 「トレーニング開始」を実行する,
  Then 新しいセッションが作成され、`started_at` に現在時刻、`workout_date`
  にクライアントローカル日付が入った状態で記録画面に遷移する
- [ ] Given 同じ日にすでにセッションが 1 件ある, When もう一度「トレーニング
  開始」する, Then 同じ `workout_date` の 2 つ目のセッションが作られる
  （二部練が表現できる）
- [ ] Given 開始済みのセッション画面, When 種目を 1 つ選択し種目ブロックを
  追加する, Then 空の種目ブロックがセッション配下に追加される
- [ ] Given 種目ブロックが追加された, When 種目の **計測項目セット** に
  応じた入力フィールドでセットの数値を入力する, Then セットがブロックに追加される
  - `reps` を持つ種目: 回数 (整数) を入力
  - `weight` を持つ種目: 重量 (kg, 符号付き) を入力。UI 上は「加重 / 補助」
    トグルで内部的に符号を反転して保存する
  - `time` を持つ種目: 秒数 (整数) を入力
  - 必須計測項目が未入力ならセット保存は拒否される
  - 任意計測項目は省略可
  - 上記いずれにも、（任意）セット間インターバル秒数を付けられる
- [ ] Given セットを追加した, When 順序を入れ替える, Then 並び順が更新される
- [ ] Given 同じセッション内で同じ種目を 2 回取り組んだ, When 同じ種目の
  ブロックをもう一度追加する, Then 同セッション内に 2 つ目の種目ブロックが
  作られる（マージしない）
- [ ] Given 入力した数値が許容範囲外（上限超過、整数フィールドに小数等）,
  When サブミットする, Then 検証エラーで拒否される
- [ ] Given 何も入力されていない空のセットを追加する, When 保存する,
  Then 空セットは破棄される（保存しない）
- [ ] Given セッション中, When 「セッション終了」を実行する, Then `ended_at`
  に現在時刻が入り、編集モードを抜ける（ただし後から編集は可能）
- [ ] Given セッション終了操作をしないまま離脱する, When 後日見る,
  Then `ended_at` は NULL のまま残り、機能上の問題は発生しない
- [ ] Given 過去日付の記録を作りたい, When 新規セッション作成時に
  `workout_date` を過去日に変更する, Then 任意の過去日付でセッションを
  作成できる（未来日不可）
- [ ] Given セッションの `workout_date` を後から編集する, When 別の日付を
  指定する, Then DATE 型として永続化される

**境界条件**:

- `workout_date` 未来日付: 不可（クライアントのローカル日付より未来は拒否）
- `workout_date` 過去日付の下限: 制約なし（実用上は数十年スパン）
- 同一日付のセッション数: 上限なし
- 1 セッションの種目ブロック数: 上限なし
- 1 種目ブロックのセット数: 50 まで
- `started_at` > `ended_at` となる編集: サーバー側で拒否

### US-8. 過去の記録を編集・削除する

> As a ユーザー,
> I want to 入力ミスや事後修正に対応するため、過去の記録を編集・削除できる,
> so that 履歴を正確な状態に保てる.

**受け入れ基準**:

- [ ] Given 過去のセッションを開く, When `workout_date` / `note` /
  種目ブロックの並び順 / 各セットの数値を編集する, Then 変更が即時保存される
- [ ] Given 種目ブロックを削除する, When 確認ダイアログで承認する,
  Then ブロックとその全セットが削除される
- [ ] Given セッションそのものを削除する, When 確認ダイアログで承認する,
  Then セッション配下の全種目ブロック・セットが削除される
- [ ] Given 削除を実行した, When 一覧を再読込する, Then その記録は完全に
  消えており復元手段はない
- [ ] Given 編集中に種目ブロックの種目を別の種目に差し替える,
  When 差し替え先の種目の **計測項目セット** が現在記録されているセットの
  値と矛盾する（必須項目が欠ける、解釈不能な項目が残る）, Then 差し替えは
  拒否される。完全に互換な場合のみ許可（典型的にはブロックを削除して
  作り直す動線になる）

### US-9. 日付別ビューでトレーニング履歴を見る

> As a ユーザー,
> I want to カレンダーや日付降順リストで、いつトレーニングしたか／その日
> 何をやったかを俯瞰したい,
> so that 全体ペースや空白期間を把握できる.

**受け入れ基準**:

- [ ] Given ログイン中, When 履歴画面 (日付ビュー) を開く, Then 自分の
  セッションが日付の新しい順に並んで表示される
- [ ] Given 日付を選択する, When 詳細を開く, Then その日の全種目ブロックと
  各セットの数値がまとめて見える
- [ ] Given セッションが 1 件もない日付, When その日付を見る, Then 空状態
  メッセージが表示される
- [ ] Given セッションが多数 (例: 50 件) 存在する, When スクロールする,
  Then ページネーションまたは仮想スクロールで遅延なく閲覧できる

**境界条件**:

- 一覧 API のページサイズ: 1 ページ 30 セッション（仮置き）

### US-10. 種目別ビュー（プログレッションツリー集約対応）で履歴を見る

> As a ユーザー,
> I want to 特定の種目／ゴール種目を選んで、その種目の過去の取り組みを
> 時系列で見たい,
> so that 次回トレーニングの重量・回数の目安を決められる／スキル進捗を
> 振り返れる.

**受け入れ基準**:

- [ ] Given 種目一覧から種目を 1 つ選ぶ, When 種目別履歴を開く,
  Then その種目に紐づく過去の種目ブロックが新しい順に並ぶ
- [ ] Given 種目別履歴を開いた, When 一番上を見る, Then 「前回の数値」
  （直近の種目ブロックのセット内容）が他と区別された見せ方で表示される
- [ ] Given 開いた種目に **子（プログレッション）が存在する**, When 「ツリー
  まとめて表示」モードに切り替える, Then その種目自身およびすべての子孫
  種目に対する記録が時系列で混ざって表示される。各エントリーには「どの
  種目に対する記録か」が明示される
- [ ] Given ツリー集約モード, When 並びを見る, Then 「ある段階の種目を
  最後に行った日付」と「次の段階の種目を最初に行った日付」が分かる
  （= いつ次のステップに上がったかが追える）
- [ ] Given 子のない種目（リーフ）, When ツリー集約モードを試す,
  Then 自分のみの履歴と等価になる（モード切替の意味はない）

**境界条件**:

- ツリー集約モードでは **数値の合算は行わない**（reps と秒数を足せない
  ため）。あくまで時系列の混合表示
- ページサイズ: 1 ページ 50 種目ブロック（仮置き）

## 画面仕様（概要レベル）

ビジュアルデザインは扱わず、表示項目と状態のみ定義する。レイアウトは
設計フェーズで詰める。

### S-1. サインアップ画面

- 表示項目: ユーザー ID 入力、パスワード入力、サブミット、ログイン画面への
  リンク
- 状態: 初期 / 入力中 / 検証エラー / 送信中 / サーバーエラー
- バリデーション: クライアントでも長さ・文字種を即時チェック

### S-2. ログイン画面

- 表示項目: ユーザー ID、パスワード、サブミット、サインアップへのリンク
- 状態: 初期 / 検証エラー / 送信中 / 認証失敗 / レート制限超過

### S-3. セッション編集画面（メイン）

- 表示項目: `workout_date`（編集可）、`started_at` / `ended_at`、`note`、
  種目ブロックとセットの一覧、「種目を追加」ボタン、「セッション終了」ボタン
- 操作: 「トレーニング開始」で新規セッションを作成 → 種目選択 → 種目ブロック
  追加 → 計測項目セットに応じたセット入力フォーム表示 → セット保存 →
  「セッション終了」または離脱
- 状態: 新規未開始 / 開始済み（記録中） / 終了済み（編集モード）/ 保存中 /
  保存エラー
- 計測項目別 UI:
  - `weight` 入力欄には「加重 / 補助」トグル（内部的に符号反転）
  - 必須計測項目は赤マーク、任意は省略可

### S-4. 種目選択モーダル

- 表示項目: 自分の種目とプリセットを混在した検索可能なリスト、種目タイプ、
  プログレッションツリーの親（あれば）
- 操作: 検索フィルター、タップで選択
- 状態: 検索クエリなし / 検索中 / マッチなし

### S-5. 種目管理画面

- 表示項目: 自分の種目とプリセットの一覧、各種目の種目タイプ、追加ボタン、
  プログレッションツリーの可視化（子をぶら下げて表示）
- 操作: 種目追加 / 編集 / 削除（自分のみ）/ プリセットの複製 / 親の変更
- 状態: 一覧表示 / 編集モーダル / 削除確認モーダル / カスケード削除警告

### S-6. 履歴画面（日付ビュー）

- 表示項目: 日付降順のセッション一覧、各セッションの種目ブロック数サマリ
- 操作: 日付タップでセッション詳細へ
- 状態: データなし / ロード中 / 表示 / 追加読み込み中

### S-7. 履歴画面（種目別ビュー）

- 表示項目: 種目名、前回の数値、種目ブロック時系列、ツリー集約モード切替
  （子があるときのみ）
- 操作: モード切替、ページネーション
- 状態: 子あり / 子なし、データなし、ツリー集約モード ON / OFF
